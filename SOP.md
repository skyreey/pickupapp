# 项目开发 SOP v3

> 适用范围：Web、移动App、小程序、桌面端、浏览器插件 — 所有软件项目
> 核心原则：**先想清楚再动手、每个阶段有产出物、不跳步、可中断可恢复、每阶段有验收条件**

---

## 总览：11阶段闭环

```
阶段0: BRD快速决策(1h)       → 值不值得做？Go/No-Go
阶段1: PRD产品定调(2-3h)     → 完整需求文档 + 安全架构
阶段2: 技术选型&初始化(1h)   → 技术栈决策 + 项目骨架 + 三套环境
阶段3: UI/UX原型(2-3h)       → 高保真原型 + 原型验证
阶段4: API&数据库设计(1-2h)  → OpenAPI + DB Schema
阶段5: 开发执行(7-14d)       → 按平台分轨 + 同步埋点
阶段6: 安全加固(1-2d)        → 八卦编码 + 混淆 + 平台加固
阶段7: 质量检查(2-3d)        → 两轮测试 + 代码审查
阶段8: 预发布验证(0.5d)      → Staging 闸门 ⚠️ 不可跳
阶段9: 上线发布(1d)          → 部署 + 回滚方案 + 30m监控
阶段10: 持续迭代             → 反馈分析 → 回到阶段1
```

---

## 阶段0：BRD 快速决策（1小时，不可跳过）

**说什么：** 「这个想法值不值得做」「帮我做BRD调研」

**干什么：** 快速判断一个想法是否值得投入开发。只做 Go/No-Go 决策，不写完整需求文档。

**6步快速调研：**

1. **市场判断** — 赛道规模多大？时机对吗？（太早/正好/太晚）
2. **竞品扫一眼** — Top 3-5 竞品，核心功能对比，优劣势各3条
3. **用户验证** — 目标用户1-2个画像，痛点排序，当前替代方案是什么
4. **商业模式** — 怎么赚钱？竞品定价区间？一个人+AI能不能做？
5. **风险评估** — 技术壁垒、资源瓶颈、核心风险Top 3
6. **Go/No-Go 建议** — 综合打分（1-10），Go 则给出 MVP 范围建议，NoGo 说明原因

**产出：** `docs/BRD.md`

**验收：**
- 6个章节齐全
- Go/No-Go 有明确结论和理由
- 竞品至少写了3条优劣势

**⚠️ No-Go 就停在这里。** 及时止损比勉强推进省十倍时间。

## 阶段1：PRD 产品需求文档（2-3小时）

**说什么：** 「写PRD」「梳理需求」「产品规划」

**干什么：** 基于 BRD 的 Go 决定，展开为完整产品需求文档。

**PRD 必须覆盖：**

- 产品定位、目标平台列表（Web/iOS/Android/小程序/桌面/插件）
- 用户画像（2-3个）与核心使用场景
- 核心功能详述（用户故事格式 + 验收标准 + 优先级）
- MVP 范围定义
- **安全架构初案**：认证方式（JWT/OAuth2）、权限模型（RBAC/ABAC）、敏感数据加密策略、API 防滥用策略
- 非功能需求（性能指标、并发要求、合规要求）
- 数据埋点事件定义（每个核心功能对应的埋点事件）
- 北极星指标

**产出：**
- `docs/PRD.md` — 完整产品需求文档
- `docs/Roadmap.md` — 版本路线图（MVP→v2.0）
- `docs/User_Story_Map.md` — 用户故事地图（可选，复杂项目建议做）
- `docs/Metrics_Framework.md` — 指标框架 + 埋点事件清单

**验收：**
- 目标平台列表明确（后续所有设计决策以此为调度依据）
- 每个核心功能有可衡量的验收标准
- 安全架构初案覆盖了认证、授权、加密三个维度
- 埋点事件清单完整（没有这个，阶段10无数据可分析）

**移动端特别说明：** 如果目标平台包含 iOS 或 Android，PRD 中需明确设计风格倾向（iOS 风格优先）。

## 阶段2：技术选型 & 项目初始化（1小时）

**说什么：** 「初始化项目」「配项目」

**干什么：** 技术栈决策 + 项目骨架搭建。这是不可逆决策，慎重。

**2.1 技术选型评审**

每个维度至少列 2 个候选方案，按以下维度对比后选定：

- 性能需求匹配度
- 与 AI 协作的友好程度（社区资源多、文档好 = AI 生成代码质量高）
- 生态成熟度
- License 合规
- 学习成本

**常见推荐（不是强制，根据项目选择）：**

| 维度 | 首选 | 备选 |
|------|------|------|
| Web 前端 | Next.js | Nuxt, Remix |
| 移动端 | Flutter | React Native |
| 桌面端 | Electron | Tauri |
| 小程序 | Taro | uni-app |
| 后端 | Node.js/Nest.js | Python/FastAPI, Go |
| 数据库 | PostgreSQL | MySQL, MongoDB |

选定后写入 `docs/Tech_Stack.md`，**不可随意更改**。开发中途换框架的重置成本足以毁掉项目进度。

**2.2 搭建项目骨架**

```
对Codex说：「初始化项目」

Codex 自动完成：
- 创建 AGENTS.md（项目规则文件，Codex 读取后理解项目上下文）
- 创建标准目录结构（包含 dev/staging/prod 三套环境配置骨架）
- 初始化 Git 仓库
```

**产出：**
- `docs/Tech_Stack.md` — 技术选型决策记录及理由
- `AGENTS.md` — 项目规则文件
- 标准项目目录结构
- Git 仓库初始化

**验收：**
- 技术选型有明确的选型和备选记录，每个维度有选型理由
- 项目目录结构包含所有标准子目录

## 阶段3：UI/UX 原型设计（2-3小时）

**说什么：** 「画原型」「出设计稿」「做界面」

**干什么：** 在写代码之前先把所有核心界面画出来，用浏览器直接预览。

**Codex 自动根据 PRD 的目标平台生成对应风格的界面原型：**

| 目标平台 | 原型样式 |
|----------|----------|
| Web | 浏览器窗口模拟 |
| iOS | iPhone 设备外壳 + Apple HIG 风格 |
| Android | Pixel 设备外壳 + Material Design 3 |
| 小程序 | 微信导航栏 + 胶囊按钮 |
| 桌面端 | 系统窗口标题栏 |
| 浏览器插件 | Popup 弹窗或 Options 页面 |

**设计要求：**
- 暗黑主题优先
- 使用 Unsplash/Pexels 真实图片，禁止占位符
- hover / active / focus / disabled 四种交互状态必须完整
- 所有核心界面通过 `index.html` 以 iframe 平铺展示，一屏看完

**产出：** `design/prototypes/index.html`（浏览器直接打开即可预览全部界面）

**⚠️ 原型回退机制：** 原型验证过程中如果发现 PRD 的功能假设不成立（交互实现不了、用户流程有死胡同、设计规范与功能冲突），**必须回到阶段1修改 PRD**。原型阶段发现问题比代码写完再改便宜 10 倍。

**验收：**
- `index.html` 浏览器打开后能看到所有核心界面的完整预览
- 没有占位符图片，所有交互状态可演示
- 原型中未发现 PRD 层面的逻辑硬伤（否则回到阶段1）

---

## 阶段4：API & 数据库设计（1-2小时）

**说什么：** 「设计 API」「设计数据库」

**干什么：** 原型验证通过后，设计后端接口和数据模型。**必须在阶段3原型验证通过后才进入这一步**——原型可能已经改过 PRD，基于过时的需求设计 API 是浪费时间。

**4.1 API 接口定义**

**产出：** `backend_service/API_Spec.md`
- OpenAPI 3.0 格式
- 所有 Endpoint 的请求/响应定义
- 必须包含认证方案（JWT/OAuth2）的接口层面描述
- 所有正常和异常响应码的 Schema 定义

**4.2 数据库设计**

**产出：** `backend_service/DB_Schema.md`
- 数据表结构（列名、类型、约束、索引）
- 核心实体关系描述
- 迁移策略约定（命名规则、执行顺序、回滚方式）

**验收：**
- API 文档覆盖 PRD 中所有核心功能的接口需求
- 认证方案在 API 层面有完整的接口描述
- 数据库设计支持 PRD 中定义的数据需求

---

## 阶段5：开发执行（7-14天）

**说什么：** 「拆分开发计划」「执行第X阶段」

**5.1 拆分任务**

Codex 将 PRD 拆分为多个 Phase，每个 Phase 对应一组可独立验收的功能模块。输出 `docs/Roadmap.md`。

**5.2 开发顺序（按项目类型判断，不搞铁律）**

```
UI 优先型项目（设计驱动，交互复杂的 App/工具）：
  原型 → 客户端 UI → 后端 API → 对接

逻辑优先型项目（数据结构复杂，交互简单的后端/API）：
  PRD → 后端 API → 客户端 UI → 对接

不确定选哪种时，默认走 UI 优先——先看到界面更容易发现需求问题。
```

**5.3 多平台分轨**

各平台共享 `backend_service/API_Spec.md`，各自独立开发：

| 平台 | 技术栈 | 目录 |
|------|--------|------|
| Web | Next.js | `web_client/` |
| iOS/Android | Flutter | `mobile_client_flutter/` |
| Windows/macOS | Electron | `desktop_client_electron/` |
| 微信/支付宝小程序 | Taro | `mini_program_taro/` |
| Chrome/Firefox插件 | Manifest V3 | `browser_extension/` |

**5.4 开发期间同步埋点** ⚠️

每个功能模块开发时同步加入数据埋点。事件定义从阶段1的 `docs/Metrics_Framework.md` 提取。没有埋点，阶段10就没有数据可分析。

**5.5 日常操作**

每完成一个 Phase：
1. 打开浏览器或模拟器查看效果
2. 不满意直接说「把XX改成YY」
3. 中断后重新打开说「继续上次」可接续
4. 遇到 Bug 直接贴错误信息

**验收：**
- 每个平台客户端核心功能可完整走通
- 埋点事件按 Metrics_Framework 全部加入
- 中断后可正常恢复开发

## 阶段6：安全加固与防破解（1-2天）

> 安全架构（认证、授权、加密策略）已在阶段1和阶段4完成设计。本阶段只做**运行时加固**：混淆、反调试、签名校验、防篡改。

**说什么：** 「加固代码」「防破解设置」

### 第一层：八卦基元表编码

用八卦符号 `☰☱☲☳☴☵☶☷` 映射 0-7，编译时编码核心常量和跳转表，运行时通过内建查表还原。逆向者在二进制中看到的是无意义的三位编码。实现难度低。

| 卦名 | 符号 | 值 | 编码用途 |
|------|------|-----|----------|
| 乾 | ☰ | 0 | 数据块起始标记 |
| 兑 | ☱ | 1 | 整数常量编码 |
| 離 | ☲ | 2 | 字符串索引 |
| 震 | ☳ | 3 | 函数跳转偏移 |
| 巽 | ☴ | 4 | 分支条件编码 |
| 坎 | ☵ | 5 | 错误码映射 |
| 艮 | ☶ | 6 | 权限标记 |
| 坤 | ☷ | 7 | 数据块结束标记 |

### 第二层：标准代码混淆

使用成熟现成工具，不造轮子：

| 平台 | 工具 |
|------|------|
| JavaScript/TypeScript | `javascript-obfuscator` |
| C/C++/ObjC (iOS/macOS) | Obfuscator-LLVM |
| Java/Kotlin (Android) | ProGuard / R8 |
| Dart (Flutter) | `dart_obfuscator` 或 `--obfuscate` |

统一要求：控制流平坦化 + 字符串 XOR 加密 + Source Map 不上传生产。

### 第三层：平台原生加固

| 平台 | 必做项 |
|------|--------|
| Web | JS 混淆 + 反调试检测 + API Key 不在前端明文出现 |
| iOS | 二进制符号剥离 + 越狱检测 + 反 Frida 注入 |
| Android | APK 签名校验 + 反 Xposed + So 库加固 |
| 桌面端 | ASAR 加密 (Electron) / 二进制加壳 (原生) |
| 小程序 | 代码分包加密 + 请求签名防抓包重放 |
| 浏览器插件 | Manifest 权限最小化 + Content Script 混淆 |

### 密码学底线（所有平台通用）

- API 请求附带 HMAC-SHA256 签名 + 时间戳，服务端验证，60 秒窗口防重放
- 发布包用 RSA-2048 签名，启动前校验完整性
- 检测到篡改：不立即崩溃，静默降级或禁用核心功能

**产出：**
- `security/hardening_plan.md` — 加固方案及八卦编码规则
- `security/crypto_config.md` — 密钥管理和签名配置

**验收：**
- 加固后的产物 grep 不到明文 API Key
- 篡改安装包后核心功能不可用
- 反调试机制正常触发

## 阶段7：质量检查（2-3天）

**说什么：** 「写测试用例」「审查代码」「测试总结」

### 7.1 测试文档

**产出：**
- `test/Test_Plan.md` — 测试范围、策略、环境、风险评估
- `test/Test_Cases.md` — 按模块组织的测试用例，覆盖正向流程、边界条件、异常输入

多平台项目自动按平台分别生成检查项。

### 7.2 测试执行（至少两轮）

```
第一轮：执行全部用例 → 汇总Bug → 集中修复
第二轮：回归测试（重测第一轮Bug + 核心Happy Path）→ 确认修复 → 测试总结
```

### 7.3 Bug 追踪

发现问题说「记个Bug」，自动填入：Bug ID + 标题 + 严重程度 + 环境 + 重现步骤 + 实际vs预期。数据写入 `test/Bug_Report.csv`。

### 7.4 代码审查

说「审查代码」，检查：代码规范、安全漏洞（XSS/SQL注入/敏感信息暴露）、性能风险点、依赖安全（已知CVE）。

### 7.5 测试总结

说「测试总结」，产出 `test/Test_Report.md`，包含执行概况、缺陷统计、遗留风险评估、发布建议（建议发布/有条件发布/不建议发布）。

**验收：**
- 所有致命和严重 Bug 已修复
- 测试报告结论为「建议发布」
- 两轮测试均已完成

---

## 阶段8：预发布验证（半天）

**⚠️ 不可跳过。** 上线前的最后一道闸门。

**说什么：** 「预发布验证」「staging验证」

**5步验证：**
1. 部署当前版本到 staging 环境
2. 执行核心测试用例（阶段7的 Happy Path + 之前出过Bug的用例）
3. 验证数据库迁移脚本在类生产数据上正常运行
4. 验证第三方服务集成（支付、推送、邮件等）在 staging 通路正常
5. 通过后标记版本为「可发布」，进入阶段9

**不通过则退回阶段7修复，不允许直接跳到阶段9。**

**验收：**
- Staging 环境核心功能可用
- 迁移脚本无错误执行
- 第三方服务集成验证通过

---

## 阶段9：上线发布（1天）

**说什么：** 「帮我部署」「发布到线上」

**部署流程：**
1. 说「帮我部署」→ 生成 CI/CD 配置、Dockerfile
2. 说「发布到线上」→ 执行部署
3. 验证线上环境：核心功能可用、监控报警已配置
4. **生成回滚方案**：记录版本号、数据库迁移号、一键回滚命令

**产出：**
- `deploy/` — 部署配置文件及部署报告（含回滚步骤）
- `monitoring/` — 监控告警配置说明
- 线上访问 URL

**⚠️ 每次部署必须携带可执行的一键回滚方案。** 上线后 30 分钟内持续监控错误率和关键指标，异常立即回滚。

**移动App/小程序/插件的发布** — 走各自应用商店或平台的分发流程，参考对应目录下的 `BUILD.md` 和 `PUBLISH.md`。

**验收：**
- 线上环境核心功能可用
- 监控报警已配置且正常触发
- 回滚方案已记录并可执行

---

## 阶段10：持续迭代（循环）

**说什么：** 「分析用户反馈」「规划下个版本」

**反馈收集：** 从各平台（App Store 评论、反馈表单、应用内埋点）汇总数据，产出报告：
- 用户情绪分布（积极/中立/消极比例）
- 高频问题与痛点排序
- 新功能请求热度排名
- 性能与可用性问题

**迭代规划：** 结合反馈分析结果更新 PRD，标注新增/修改的功能点，从阶段3开始新迭代。

**迭代循环：**
```
反馈分析 → 更新PRD → 原型(如有UI变更) → API更新(如需要) → 开发新Phase → 加固 → 测试 → staging → 发布
```

**应急调试：** 遇到棘手 Bug 直接贴错误信息和日志，输出：根因 → 修复方案 → 执行步骤 → 原理说明。

---

## 完整闭环

```
阶段0: BRD(Go) → 阶段1: PRD+安全架构 → 阶段2: 技术选型+初始化 → 阶段3: 原型验证 → 
阶段4: API+DB设计 → 阶段5: 开发(同步埋点) → 阶段6: 安全加固 → 
阶段7: 测试(两轮) → 阶段8: Staging闸门 → 阶段9: 发布(带回滚) → 阶段10: 反馈 → 回到阶段1 ...
```

| 「拆分开发计划」 | 阶段5 | Roadmap + Phase 规划 |
| 「执行第X阶段」「继续上次」 | 阶段5 | 逐阶段开发 |

### 遇到什么问题 → 找哪个阶段

| 问题 | 答案 |
|------|------|
| 不知道值不值得做 | 阶段0 — BRD 调研 |
| 不知道做成什么样 | 阶段3 — UI/UX 原型 |
| 不知道怎么实现 | 阶段4+5 — API设计+开发 |
| 代码报错了 | 贴错误信息就行 |
| 功能不正常 | 阶段7 — 记Bug+修复 |
| 怕被破解 | 阶段6 — 三层加固 |
| 不知道该不该上线 | 阶段8 — Staging 验证 |
| 不知道怎么部署 | 阶段9 — 部署+回滚 |
| 用户反馈看不懂 | 阶段10 — 反馈分析 |
