| # | 场景 | 覆盖 | 缺口 |

| 空白场景 | 严重度 | 说明 |
|----------|--------|------|
| **物流轨迹查询** | 🔴 P0 | PRD 声称"✅ 完成"但代码里无任何物流 API 调用。`TrackingApiResult` 接口存在但无实现。`tracker-scheduler.ts` 只做本地 deadline 检查，不查物流。详情页无轨迹展示代码。 |

## 四、下一步开发建议（按优先级）

### 🔴 P0 — 阻断发布

**1. 实现物流轨迹查询**
这是 PRD 声称完成但实际为 0 的功能。建议：
- 接入快递鸟/快递100 API（免费额度够个人项目）
- 在 `detail/[id].tsx` 增加轨迹时间线组件
- `tracker-scheduler.ts` 增加定时查询逻辑（现有调度框架可复用）
- 自动签收检测：API 返回"已签收"→ `markAsPickedUp`

**2. 部署激活服务端**
`API_BASE` 仍是 `example.com` 占位域名。需要：
- 搭建 `/api/verify`、`/api/sync-membership`、`/api/restore-membership` 端点
- 设置 `EXPO_PUBLIC_API_BASE` 环境变量
- 否则 Pro 付费体系形同虚设

### 🟠 P1 — 发布前应做

**3. 完善 Onboarding 权限引导**
当前 `onboarding.tsx` 只有基础引导（`router.replace`），缺少：
- 首次启动时引导授予 SMS 权限
- 引导开启通知使用权（跳转系统设置）
- 权限拒绝后的重新引导路径

**4. OCR 真机测试**
Roadmap 标 P0 待做。ML Kit OCR 在不同机型/Android 版本上表现差异大，需验证：
- 截图识别准确率
- 多图批量性能
- 内存占用

**5. 全功能回归测试**
213 个单元测试覆盖了解析层，但缺少：
- UI 交互测试（列表筛选、批量操作、导航）
- 数据库集成测试（迁移、事务、去重）
- 端到端测试（SMS→解析→入库→列表展示）

### 🟡 P2 — 发布后迭代

**6. 物流异常状态处理**
增加 `returned`（已退回）、`failed`（派送失败）状态，对应通知和 UI。

**7. OCR 人工校正**
OCR 解析失败时提供手动编辑入口，而非直接丢弃。

**8. 远程配置服务端**
`remote-config.ts` 已实现 SMS 规则热更新框架，部署服务端后可在不发版的情况下修复解析规则。

**9. 埋点与数据分析**
BRD 提到"开发期间同步埋点"，但代码里只有 `perf-tracker.ts`（性能），无业务埋点（取件率、识别准确率、Pro 转化率）。

### ⚪ P3 — 远期

**10. 桌面挂件重写**
`expo-pickup-widget` 模块代码存在但 README 标注"已移除"，v2.0 可重写 AppWidget。

**11. iOS 版**
等 Apple 开放 SMS 读取权限（短期无望）。

## 五、总结

取件通的核心价值链——**自动识别 → 聚合管理 → 提醒取件**——已经完整实现，数据采集和解析引擎是项目的强项，覆盖了 SMS/通知/OCR/手动四个入口。包裹管理、提醒通知、家庭代取等功能也都落地了。

最大的问题是**物流轨迹查询完全是空白**——PRD 和 Roadmap 都标了"✅ 完成"但代码里只有接口定义没有实现。这是需要优先补上的功能缺口。其次是激活服务端和远程配置服务端都还没部署，代码写好了但没有后端支撑。

建议发布顺序：补物流轨迹 → 部署激活服务端 → 完善 Onboarding → OCR 真机测试 → 回归测试 → 上线。
