## 一、项目概况

取件通是一款 Android 快递包裹自动追踪助手，核心价值在于"**无需手动输入，自动识别取件信息**"。用户只需授予 SMS 读取和通知监听权限，App 即可自动扫描短信和购物通知，提取取件码、快递公司、站点地址等信息，形成包裹管理列表。

| 项目 | 内容 |
|------|------|
| 立项时间 | 2026-05-22 |
| 当前评估 | ~92% 完成 |
| 目标用户 | 中国 Android 用户（高频网购人群） |
| 技术路线 | Expo 托管 → Bare Workflow |
| 开发设备 | Huawei PLA_AL10 (Android 10) |

## 三、架构决策分析

### 3.1 为什么选择 Bare Workflow 而非纯 Expo Go？

**原因**：需要原生 SMS/NOTIFICATION_LISTENER 权限，Expo Go 不支持自定义原生模块。

**代价**：
- 失去了 Expo Go 的快速热重载
- 需要维护 `android/` 原生工程
- 每次 JS 改动需 `npx expo run:android` 重新编译

**缓解措施**：
- 使用 `debuggableVariants = []` 确保 Debug 构建正确嵌入 JS Bundle
- 关键模块（SMS/通知）的逻辑集中在 JS 侧，减少原生代码迭代

### 3.2 SMS 双通道设计

```
实时通道：SmsReceiver (Broadcast) → JS Event → handleIncomingSms()
历史通道：内容提供者扫描 → scanSmsInbox() → handleHistoricalSms()
```

- 实时通道用于新短信即时响应
- 历史通道用于启动时全量扫描
- `hasScannedSms` 标志防止重复扫描

### 3.3 去重策略

三级去重确保不创建重复包裹：

| 级别 | 手段 | 范围 |
|------|------|------|
| 1 | `processedHashes` Set | 单次会话，最多 200 条 |
| 2 | `getPackageByPickupCode()` | 数据库全局 |
| 3 | `deduplicateAllPickupCodes()` | 同取件码合并 |

### 3.4 通知 vs SMS 优先级

设计上 SMS 是主要入口，通知作为补充：

- **SMS**：所有快递公司都会发送取件短信，格式稳定，解析准确率高
- **通知**：仅部分 App 推送，格式多样，但能获取商品名和单号

**匹配策略**：通知创建的包裹（无取件码）→ 后续收到 SMS 取件码 → `findMatchingPackage` 自动关联

### 3.5 设置持久化方案

| 方案 | 存储位置 | 适用场景 |
|------|----------|----------|
| expo-sqlite | packages.db | 包裹数据 |
| SharedPreferences | 原生模块 | 设置项（历史天数/扫描标志/监控开关） |
| 内存变量 | settings-store.ts | 启动时加载，运行时快速访问 |

选择 SharedPreferences 而非 AsyncStorage 的原因：原生模块（SmsReceiver、NotificationListenerService）在 JS 层未加载时也需要读取设置状态。

## 四、数据库设计

### 4.1 Schema

```sql
CREATE TABLE packages (
  id TEXT PRIMARY KEY,          -- 唯一 ID (UUID)
  trackingNumber TEXT,          -- 快递单号
  carrier TEXT,                 -- 快递公司代号
  carrierName TEXT,             -- 快递公司名称
  orderSource TEXT,             -- 订单来源 App
  productName TEXT,             -- 商品名称
  pickupCode TEXT,              -- 取件码
  pickupAddress TEXT,           -- 取件地址
  pickupPointName TEXT,         -- 站点名称
  pickupPointPhone TEXT,        -- 站点电话
  businessHours TEXT,           -- 营业时间
  currentStatus TEXT,           -- 当前状态
  statusUpdatedAt INTEGER,      -- 状态更新时间 (unix ms)
  source TEXT,                  -- 来源 (sms/notification/manual/ocr)
  createdAt INTEGER,            -- 创建时间
  pickedUpAt INTEGER,           -- 取件时间
  notes TEXT                    -- 用户备注
);
```

### 4.2 状态枚举

```typescript
type PackageStatus = 
  | 'pending'      // 待发货
  | 'shipped'      // 已发货
  | 'in_transit'   // 运输中
  | 'arrived'      // 已到站（但未发取件码）
  | 'stored'       // 已到站可取件（有取件码）
  | 'picked_up'    // 已取件
  | 'error';       // 异常
```

### 4.3 状态迁移

```
pending → shipped → in_transit → arrived → stored → picked_up
                                          ↘  (自动签收) ↗
error → 任意状态 (重试后)
```

## 五、原生模块详细说明

### 5.1 expo-sms-reader

| 文件 | 职责 |
|------|------|
| `ExpoSmsReaderModule.kt` | Expo 桥接，暴露 scanSmsInbox/getPendingSms/startListening/stopListening 等方法 |
| `SmsReceiver.kt` | BroadcastReceiver，监听 `android.provider.Telephony.SMS_RECEIVED` |

**架构**：
- `SmsReceiver` 在 AndroidManifest 中静态注册（开机自启）
- 收到 SMS → 写入 `pendingSms` 队列 (ConcurrentLinkedQueue)
- JS 层通过 `getPendingSms()` 拉取队列
- 前台时使用实时 Listener 直接推送事件到 JS

### 5.2 expo-notification-reader

| 文件 | 职责 |
|------|------|
| `ExpoNotificationReaderModule.kt` | Expo 桥接 + 14 App 监控开关 + SharedPreferences 通用设置 |
| `PickupNotificationService.kt` | NotificationListenerService，实时监听 + 启动快照 |

**监控目标（14 个 App）**：

### 5.3 expo-ocr-reader

- 基于 Google ML Kit Text Recognition (Chinese)
- 提供 `recognizeText(uri: String): OcrResult` 方法
- 输入图片 URI → 输出识别文字 + 置信度
- JS 层 `ocr-parser.ts` 对识别结果进行二次解析（提取单号/公司/状态）

### 5.4 expo-app-scanner

- 使用 `PackageManager.getInstalledApplications()` 
- 对比预设购物 App 列表 → 返回已安装项 + 首次安装时间
- 用于设置页"已检测到的购物 App"展示

### 5.5 expo-gallery-launcher

- 封装系统 `ActivityResultContracts.GetMultipleContents()`
- 实现多选图片（MIME type: image/*）
- 选择后返回 `content://` URI 列表

## 六、性能优化记录

### 6.1 APK 体积优化（从 ~210MB 到 ~24MB）

| 优化项 | 操作 | 效果 |
|--------|------|------|
| ABI 裁剪 | 仅编译 arm64-v8a | -60%~70% so 体积 |
| R8 混淆 | 启用 minifyEnabled | -20%~30% class 体积 |
| 资源压缩 | 启用 shrinkResources | 移除未使用资源 |
| 构建模式 | Debug → Release | Hermes 生产模式编译 |
| 死代码清理 | 删除 expo-pickup-widget 模块 | -5 Kotlin 文件 + 布局资源 |
| Legacy 打包 | 开启 useLegacyPackaging | Hermes/Fabric 兼容 |

### 6.2 耗电优化

| 优化项 | 操作 | 效果 |
|--------|------|------|
| 后台监听开关 | 新增全局开关，JS 层跳过扫描 | 用户可控是否监听 |
| 扫描范围 | 默认从 30 天改为 7 天 | 减少启动时的 SMS 查询量 |
| hasScannedSms | 首次扫描后设标志 | 避免每次启动重复扫描收件箱 |
| processedHashes | 内存去重 | 避免重复解析 |

**注意**：原生 SmsReceiver BroadcastReceiver 仍在 manifest 中静态注册，无法在运行时解除。JS 层忽略所有事件来达到"关闭"效果。

### 6.3 数据库优化

- 索引列（`pickupCode`, `trackingNumber`）查询加速
- 批量操作使用事务
- deduplicateAllPickupCodes 使用 DELETE + ROW_NUMBER 优化

## 十、构建与部署

### 10.1 环境要求

```powershell
# Java
set JAVA_HOME=C:\Program Files\Android\Android Studio\jbr

# Android SDK
set ANDROID_HOME=%USERPROFILE%\AppData\Local\Android\Sdk

# ADB (手动指定路径)
C:\android-sdk\platform-tools\adb.exe
```

### 10.2 构建命令

```bash
# Debug APK
npx expo run:android

# Release APK (推荐)
cd android
.\gradlew assembleRelease

# 注意：不要先执行 clean，否则 CMake codegen 产物丢失
# 如果必须 clean，先执行一次 assembleRelease 生成中间产物
```

### 10.3 安装流程

```bash
# 1. 卸载旧版本
adb uninstall com.carl.pickupapp

# 2. 安装新 APK
adb install android/app/build/outputs/apk/release/app-release.apk

# 3. 授权 SMS 权限（关键步骤！）
adb shell pm grant com.carl.pickupapp android.permission.READ_SMS

# 4. 验证权限
adb shell dumpsys package com.carl.pickupapp | findstr "READ_SMS"
```
